# Policy-24-0029 · PF Max/Min·No Load 연산/화면 정책

## 결정 사항

- PF Max/Min 연산 시 기존과 동일하게 **절대값을 취하지 않고** 연산.
- **No Load(피상전력 0) 상태의 PF 값("0.000"/"1.000")은 Max/Min 연산에 포함하지 않음.**
- No Load 관련 상황별 PF Peak 화면 정의:
  - **상황 1. No Load에서 Data Reset**: Min/Max Value는 "PF Value at No Load" 설정값(예: 1.000), Angle "No Load", Timestamp는 Reset 시점.
  - **상황 1-2. 이후 PF Value at No Load 설정 변경**(1→0): Value 0.000, Angle "No Load", Timestamp는 설정 변경 시점.
  - **상황 1-3. 이후 Load 상태로 천이**: 실측 PF Min/Max의 Value·Angle·갱신시점 표시(기존 No Load 값은 연산 제외).
  - **상황 2. Load → No Load 천이**: 기존 PF Min/Max 유지(No Load 값 연산 제외).
- 예시-A2500:

> [이미지 image 104.png] [Tobe/A2500, 상황1] HOME > POWER > PF(Max) — A/B/C/Total **"No Load" + 1.000**, Timestamp = Data Reset 시점(PF=1 설정).

> [이미지 image 105.png] [Tobe/A2500, 상황1-2] 동일 화면 — 설정 변경 후 **0.000**, Timestamp = 설정 변경 시점.

> [이미지 image 106.png] [Tobe/A2500, 상황1-3] Load 천이 후 — **Lag 0.866/0.865/0.871/0.867** 실측값·갱신시점 표시.

> [이미지 image 107.png] [Tobe/A2500, 상황2] No Load 천이 후 — 기존 PF Max(Lag 0.866 등) 유지.

- 예시-A7300:

> [이미지 image 108.png] [Tobe/A7300 METER, 상황1] HOME > POWER > PF(Max) — No Load + 1.000(파랑 테마).

> [이미지 image 109.png] [Tobe/A7300, 상황1-2] 설정 변경 후 0.000.

> [이미지 image 110.png] [Tobe/A7300, 상황1-3] Load 천이 후 Lag 0.858/0.861/0.862/0.862.

> [이미지 image 111.png] [Tobe/A7300, 상황2] No Load 천이 후 기존 값 유지.

## 결정 배경

- (2024/08/14) 1차 결정: "Invalid PF를 Max/Min 연산에 제외 + **절대값 비교 연산**" (2024/08/14 이후 출시/리뉴얼 제품 대상).

> [이미지 0acc98e2-468c-49cc-a013-6df6ec6fb551.png] [참고] 조국춘·문부기·김민수 이메일 스레드(2024-08) "PF max/min 연산" — A7300 절대값 처리 여부 문의 → 적용 제품 리스트(A2500/A3700N/A7300 등) 논의 → 절대값 연산 결정 경위.

- (2024/10/21) 추가 결정 — **절대값 정책 철회**: A2500 현황 파악에 오류가 있었고(A2500도 A3700N처럼 절대값 미사용), 모든 장치가 절대값을 취하지 않는 데다 효용성도 낮음(전력 등 다른 음수 가능 값의 Max/Min과 달라 혼동 유발).

> [이미지 image 112.png] [참고] 문부기 카톡 — "PF 절대값 처리는 PF sign off 시와 A2500 Displacement PF 연산 시뿐이며 둘 다 폐기됨. 현재 PF Min/Max 연산은 전부 있는 그대로의 값 사용. 지난 조사는 Displacement PF 코드를 보고 오인."

> [이미지 image 113.png] [참고] 조국춘 카톡(2024-10-21) — "PF 절대치 연산에 의한 max/min 정책은 철회하는 것이 타당해 보입니다."
